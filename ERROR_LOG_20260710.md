## 一、Build 階段錯誤

### #01 `requirements.txt` 版本枷釋被解凍
- **時間**：07-10 18:45 前
- **現象**：Railway Build 自動拉到最新版 `pandas==3.0.3` + `numpy==2.5.1`（Python 3.12）
- **錯誤訊息**：`ERROR: Ignored the following versions...`
- **根因**：`requirements.txt` 用 `>=` 而非鎖定版本
- **處置**：改為 `==` 固定版號（`pandas==2.2.3`, `numpy==1.26.4`, `streamlit==1.39.0`）

### #02 假依賴套件 `cache_buster_1783678781`
- **時間**：07-10 19:12 - 19:30
- **現象**：Build log 顯示 `Collecting cache_buster_1783678781` → `ERROR: No matching distribution found`
- **錯誤碼**：`ERROR: Could not find a version that satisfies the requirement cache_buster_1783678781`
- **根因**：舊版 `requirements.txt` 末尾附加了 `cache_buster_{timestamp}` 作為 cache buster，被 pip 視為套件名稱
- **處置**：清空 `requirements.txt`，重寫為乾淨版
- **備注**：GitHub API 上傳仍返回舊版，Railway build cache 卡住 15 分鐘

### #03 Railway Build Log 顯示舊版快取
- **時間**：07-10 19:12 - 19:45
- **現象**：Railway Build log 仍顯示 `streamlit>=1.29.0`, `pandas>=2.0.0`, `yfinance>=0.2.36` 等舊版無鎖版本，即使 GitHub 已更新多次
- **根因**：Railway Build cache 嚴重過期，或部署佇列未正確清空
- **處置**：
  1. 修改 `Procfile` 內容強制新 commit
  2. 修改 `.python-version` 鎖定 Python 3.11
  3. 用 GitHub API 多檔案同時上傳
  4. 最終用戶手動 Redeploy > Railway 才吃到最新 commit

## 二、Deploy / 啟動階段錯誤

### #04 `02` Bad Gateway - Port 寫死 80
- **時間**：07-10 19:31
- **現象**：用戶打開儀表板得到 `502 Bad Gateway`，`Application failed to respond`
- **錯誤訊息**：`Request ID: 3yacy_pjQjC1za3An6XIxQ`
- **根因**：`railway.toml` 寫死 `startCommand = "streamlit run dashboard.py --server.port 80 --server.headless true"`
- **影響**：Railway US West 使用動態 `$PORT`，寫死 80 導致 inbound 連線失敗
- **處置**：移除 `railway.toml` 内部 `startCommand`，統一用 `Procfile` 的 `$PORT`

### #05 `Procfile` vs `railway.toml` 啟動指令衝突
- **時間**：07-10 19:25 - 19:45
- **現象**：等多檔案同時存在時，Railway 優先使用 `railway.toml` 的 `startCommand`，忽略 `Procfile`
- **根因**：兩處同時定義，Railway 行為不一致
- **處置**：統一為單一來源—`Procfile` 定義啟動指令，`railway.toml` 只設定 `restartPolicy` 和 `healthcheck`

## INC-104: 重複執行管線造成通知轟炸（2026-07-23）

**類型：** P2（非資料錯誤，但影響使用者體驗）

## INC-105: 2026-07-23 工作颱風尾總檢討

**類型：** P3（流程問題）

**事件：** 7/23 整天修改過程中暴露多項系統性缺失。

### 缺失列表

**1. hermes/scripts/ 同步漏洞**
修改 hunter_intel.py、daily_intel.py 等腳本後，多次忘記複製到 hermes/scripts/，導致 cron 執行到舊版。無同步機制。

**2. 修復像打地鼠（patch over patch）**
一天內修了 6 個問題（負債比率、RSS 新聞、briefing、信用卡、CIO區塊、pre_push_audit），修 A 壞 B。無預先收斂。

**3. 重複跑管線洗版（INC-104 關聯）**
9 次 pipeline = 9 次資產變化對照。未先檢查本地檔案。

**4. 角色混淆（CIO vs Pro）**
使用者說 CIO 給了 Pro，說 Pro 給了 CIO。角色指派混亂。

**5. 改程式自己硬幹**
程式碼修改應給 CIO 處理，不該自己動手修。

### 改善對策
1. 建立 scripts/sync_all.py 自動同步 hermes/scripts/
2. 修復驗證只看本地檔案，不跑管線推播
3. **任務分派鐵則：自己試 2-3 次失敗 → 提議請 CIO。費用是次要，解決問題優先。**
4. 使用者說誰就給誰，不再自創角色

**狀態：** ✅ 已記錄。
